This post is the exact recipe: the checkpoint trap that costs newcomers an afternoon, the five commands that get the model loading, freshly re-measured numbers for what it actually costs in memory and time — and a hands-on attempt to serve it with SGLang-Omni — this model's official serving stack — on the same machine.Higgs TTS v3 해부해보다를 만든 모든 작업 — 36개 레이어 활성 캡처, 음성 복제 합성, 더 넓은 프로젝트의 스티어링 실험 — 은 노트북 한 대에서 돌았다. 32 GB RAM의 M1 Max 맥북 프로다. 40억 파라미터 오디오 LLM이라면 CUDA 서버가 필요할 거라 생각하기 쉽지만, 아니다. 이 글은 그 정확한 레시피다: 처음 오는 사람의 오후를 잡아먹는 체크포인트 함정, 모델을 로드시키는 명령 다섯 개, 메모리와 시간이 실제로 얼마나 드는지 오늘 다시 측정한 수치들 — 그리고 이 모델의 공식 서빙 스택인 SGLang-Omni로 같은 머신에서 직접 서빙을 시도한 실험까지.

Why it's not obvious왜 간단하지 않은가

The trap before the first token첫 토큰 전에 만나는 함정

The official repo, bosonai/higgs-audio-v3-tts-4b, ships weights and a config — but no modeling code. Point AutoModel at it and you get nothing that generates speech. The working path is the community transformers port, multimodalart/higgs-audio-v3-tts-4b-transformers: the same weights repackaged with a trust_remote_code modeling file, so plain transformers (≥5.5) can load it and call generate_speech(). Underneath sits a Qwen3-style backbone — 36 layers, hidden 2560 — with an audio tower and a 25 fps, 8-codebook codec head on top.공식 저장소 bosonai/higgs-audio-v3-tts-4b에는 가중치와 config만 있고 모델링 코드가 없다. AutoModel을 갖다 대도 음성을 생성하는 무언가는 나오지 않는다. 작동하는 경로는 커뮤니티 transformers 포트인 multimodalart/higgs-audio-v3-tts-4b-transformers다: 같은 가중치를 trust_remote_code 모델링 파일과 함께 재포장한 것으로, 순정 transformers(≥5.5)가 로드해서 generate_speech()를 부를 수 있다. 그 아래에는 Qwen3 계열 백본(36 레이어, hidden 2560)과 오디오 타워, 그리고 25 fps 8-코드북 코덱 헤드가 있다.

The choice선택

Why we skipped the official serving stack공식 서빙 스택을 건너뛴 이유

Boson AI's intended way to run Higgs Audio v3 is not "load it in transformers" — it's a serving stack: SGLang-Omni / vLLM-Omni, the audio-enabled variants of the two big LLM inference servers. Those are excellent at what they're for — streaming, batching, prefix caching, production endpoints — and they are CUDA software. We chose the plain transformers port instead, for two reasons that matter more than speed.Boson AI가 의도한 Higgs Audio v3 실행 방식은 "transformers로 로드"가 아니라 서빙 스택이다: SGLang-Omni / vLLM-Omni, 즉 양대 LLM 추론 서버의 오디오 지원 변종들. 스트리밍·배칭·프리픽스 캐싱·프로덕션 엔드포인트라는 본연의 목적에는 탁월하지만, 이들은 CUDA 소프트웨어다. 우리는 대신 순정 transformers 포트를 골랐다 — 속도보다 중요한 두 가지 이유 때문에.

Reason 1 — the part that serves TTS doesn't run on this laptop. The audio serving lives in SGLang-Omni, a companion runtime built on Triton kernels and sgl-kernel — a stack that assumes Linux + NVIDIA (every published Higgs benchmark is an H100 or H20). We didn't take that on faith: the experiment at the bottom of this post is us trying to install it anyway, error messages and all.이유 1 — TTS를 서빙하는 부분이 이 노트북에서 안 돈다. 오디오 서빙은 SGLang-Omni라는 동반 런타임에 있는데, Triton 커널·sgl-kernel 위에 선, Linux + NVIDIA를 전제하는 스택이다 (공개된 Higgs 벤치마크는 전부 H100 아니면 H20이다). 말로만 믿지 않았다: 이 글 하단의 실험이 그걸 그래도 설치해 보려 한 기록이고, 에러 메시지까지 다 담겨 있다.

Reason 2 — a server gives you audio; we needed the model's thoughts. The whole point of the research was to watch the residual stream while the model speaks. A serving stack hands you an API that returns audio — the activations live in another process, behind kernels you can't hook. In-process transformers gives you the model as a plain PyTorch module, where "looking inside" is genuinely this small:이유 2 — 서버는 음성을 주지만, 우리는 모델의 생각이 필요했다. 연구의 핵심은 모델이 말하는 동안 residual stream을 지켜보는 것이었다. 서빙 스택은 음성을 돌려주는 API를 줄 뿐 — 활성값은 다른 프로세스 안, 훅을 걸 수 없는 커널 뒤에 산다. 인프로세스 transformers에서는 모델이 평범한 PyTorch 모듈이라, "안을 들여다보기"가 정말로 이만큼 작아진다:

# the model's 36 transformer layers are just… there
layers = model.model.layers                       # Qwen3-style backbone, hidden 2560

acts = {}
def probe(layer_idx):
    def hook(module, args, out):
        h = out[0] if isinstance(out, tuple) else out    # residual stream: [1, seq, 2560] at
        acts.setdefault(layer_idx, []).append(           #   prefill, [1, 1, 2560] per decode step
            h.detach().mean(dim=(0, 1)).cpu())          # pool + off the GPU, immediately
    return hook

handles = [l.register_forward_hook(probe(i)) for i, l in enumerate(layers)]
try:
    wav = model.generate_speech(text, tok, reference_audio=ref, ...)  # hooks fire every token
finally:
    for h in handles: h.remove()                  # read-only: nothing about the model changed

That's the entire mechanism behind the other post: register_forward_hook fires at every layer for every generated token, we pool each step's output and ship it to CPU on the spot (the per-clip vector is then the mean over all steps — the real capturer keeps a running mean), and the hooks come off in a finally. No weights touched, no forward pass altered — the model never knows it was watched. Every probe, every point cloud, every steering vector in the research started as exactly this dictionary of vectors. This is what a serving stack, for all its speed, cannot give you.저것이 다른 포스트의 전체 메커니즘이다: register_forward_hook이 생성되는 토큰마다 모든 레이어에서 발화하고, 스텝마다 출력을 풀링해 그 자리에서 CPU로 보내고(클립당 벡터는 전체 스텝의 평균 — 실제 캡처러는 러닝 평균을 유지한다), 훅은 finally에서 걷어낸다. 가중치는 건드리지 않고 forward 연산도 바꾸지 않는다 — 모델은 감시당한 줄도 모른다. 연구의 모든 프로브, 모든 포인트 클라우드, 모든 스티어링 벡터가 정확히 이 벡터 딕셔너리에서 출발했다. 서빙 스택이 아무리 빨라도 줄 수 없는 것이 바로 이것이다.

The recipe레시피

Five commands to a working voice cloner음성 복제기까지 명령 다섯 개

# 1) isolated venv on top of a working MPS torch (ours: torch 2.12.0, python 3.11)
python3.11 -m venv --system-site-packages /tmp/hav
/tmp/hav/bin/pip install --no-deps \
    'numpy==1.26.4' 'transformers==5.12.1' 'tokenizers==0.22.2' \
    'huggingface_hub==1.22.0' hf_xet safetensors 'torchaudio==2.11.0'

# 2) keep TF/JAX out of the import path, let MPS fall back for unsupported ops
export USE_TF=0 USE_FLAX=0 USE_JAX=0 USE_TORCH=1 PYTORCH_ENABLE_MPS_FALLBACK=1

Then the entire Python side is this (essentially the code every capture in the research repo runs through):파이썬 쪽은 이게 전부다 (연구 저장소의 모든 캡처가 통과하는 코드와 사실상 같다):

from transformers import AutoModelForCausalLM, AutoTokenizer
import torch, soundfile as sf

REPO = "multimodalart/higgs-audio-v3-tts-4b-transformers"
tok   = AutoTokenizer.from_pretrained(REPO, trust_remote_code=True)
model = AutoModelForCausalLM.from_pretrained(
            REPO, trust_remote_code=True, dtype=torch.bfloat16
        ).to("mps").eval()                     # 12.8 s from local cache

ref, sr = sf.read("reference.wav", dtype="float32")   # the voice to clone
wav = model.generate_speech(
        "Any sentence you want, in that voice.", tok,
        reference_audio=torch.from_numpy(ref).unsqueeze(0),
        reference_sample_rate=sr,
        reference_text="…transcript of the reference clip…",
        max_new_tokens=480, temperature=0.7, top_p=0.95)
sf.write("clone.wav", wav.float().cpu().numpy(), model.config.sample_rate)  # 24 kHz
Small gotchas, so they don't cost you an afternoon: pass trust_remote_code=True to the tokenizer too — without it, transformers raises a [y/N] prompt that blocks in a terminal (headless runs print it and silently decline the tokenizer's custom code). The load report prints audio_head.weight | MISSING — benign for TTS use (that head is re-initialized and unused by generate_speech). And the first generation is slower than every one after it: the audio tower lazy-loads 527 extra weight tensors on first call.오후를 날리지 않기 위한 소소한 함정들: trust_remote_code=True는 토크나이저에도 넘겨야 한다 — 없으면 transformers가 [y/N] 프롬프트를 띄우는데, 터미널에서는 거기서 멈추고 헤드리스에서는 출력만 남긴 채 토크나이저의 커스텀 코드를 조용히 거절한다. 로드 리포트에 찍히는 audio_head.weight | MISSING은 TTS 용도로는 무해하다 (그 헤드는 재초기화되고 generate_speech는 쓰지 않는다). 그리고 첫 생성은 이후보다 느리다: 첫 호출에서 오디오 타워가 527개의 추가 가중치 텐서를 지연 로드한다.

The number that fooled us: RSS. Ask ps and the Python process claims a laughable 0.5 GB while the GPU is holding a 4B model. On Apple Silicon the weights live in unified memory pages accounted to the Metal driver, not the process — so watch torch.mps.driver_allocated_memory() (13.0 GB here), not your process monitor. The clock스톱워치

Where the seconds go — first token vs first sound초는 어디로 가는가 — 첫 토큰과 첫 소리

"How fast is it" hides two very different questions: how long until the model starts answering (time to first token), and how long until you hear something. We measured both, using the same trick as the research — a forward pre-hook on layer 0 that timestamps every generation step. The prompt here is 229 tokens (the reference audio's codec tokens + both transcripts); each decode step then produces one new audio token."얼마나 빠른가"에는 서로 다른 두 질문이 숨어 있다: 모델이 답을 시작하기까지(첫 토큰까지의 시간), 그리고 실제로 소리가 들리기까지. 둘 다 측정했다 — 연구 때와 같은 트릭으로, 0번 레이어에 forward pre-hook을 걸어 생성 스텝마다 타임스탬프를 찍었다. 여기서 프롬프트는 229 토큰(레퍼런스 오디오의 코덱 토큰 + 두 트랜스크립트)이고, 이후 디코드 스텝 하나가 오디오 토큰 하나를 만든다.

The honest catch: first token ≠ first sound. The first audio token exists 0.65 seconds in — but this transformers port only decodes tokens into a waveform after all of them are generated. So the time until you actually hear anything is the full 12.7 seconds. Nothing streams. That gap — 0.65 s of "the model has started" vs 12.7 s of silence — is precisely the problem serving stacks exist to solve, which brings us to the experiment below.정직한 반전: 첫 토큰 ≠ 첫 소리. 첫 오디오 토큰은 0.65초 시점에 이미 존재한다 — 하지만 이 transformers 포트는 토큰이 전부 생성된 뒤에야 파형으로 디코드한다. 그래서 실제로 소리가 들리기까지는 12.7초가 통째로 걸린다. 아무것도 스트리밍되지 않는다. 이 간극 — "모델은 이미 시작했다"의 0.65초와 12.7초의 침묵 — 이 바로 서빙 스택이 존재하는 이유이고, 아래 실험으로 이어진다.
The experiment실험

So, can SGLang-Omni serve it on this MacBook?그래서, SGLang-Omni로 이 맥북에서 서빙이 되나?

That 12.7-second silence begs the question. Higgs TTS v3's official serving stack, SGLang-Omni, got day-0 support for this exact model (June 2026, co-announced with Boson AI), and it streams: PCM chunks start flowing after the first codec frames. Their published single-request numbers on one H100 — mean end-to-end latency 617 ms, RTF 0.147, ~6.8× faster than realtime. The Higgs model card points you straight at it:12.7초의 침묵은 자연스레 질문을 부른다. Higgs TTS v3의 공식 서빙 스택 SGLang-Omni는 바로 이 모델의 day-0 지원을 받았고(2026년 6월, Boson AI와 공동 발표), 게다가 스트리밍한다: 첫 코덱 프레임들이 나오는 즉시 PCM 청크가 흐르기 시작한다. H100 한 장에서의 공개 단일 요청 수치 — 평균 end-to-end 지연 617 ms, RTF 0.147, 실시간보다 약 6.8배 빠름. Higgs 모델 카드는 바로 이것을 가리킨다:

# the command the Higgs cookbook gives you
sgl-omni serve --model-path bosonai/higgs-audio-v3-tts-4b --port 8000

Could we run that on the laptop? SGLang-Omni isn't on PyPI, so you install it from source. Try that on the Mac and the resolver stops before it reaches a single audio dependency:이걸 노트북에서 돌릴 수 있을까? SGLang-Omni는 PyPI에 없어서 소스로 설치한다. 맥에서 시도하면 리졸버는 오디오 의존성 하나에 닿기도 전에 멈춘다:

What Higgs-on-SGLang-Omni would actually buy — and how to get it. On supported hardware (a rented CUDA box), sgl-omni serve --model-path bosonai/higgs-tts-3-4b (the repo was renamed from higgs-audio-v3-tts-4b; the old name redirects) gives you the published H100 numbers above — but the deeper win for TTS is streaming: first PCM chunk after the first codec frames (the server defaults its first audio chunk to a single frame), against our port's generate-everything-then-vocode 12.7 s. There's also a fit with this project's exact workload: SGLang-Omni's prefix caching (RadixAttention) reuses the KV cache of a shared prompt prefix, and our 16-direction sweep reuses one reference voice per source language — each source's ~0.65 s prefill would be paid once instead of four times, and the steering α-sweep (one fixed reference across every α) would pay it exactly once. For bulk generation, the serving stack is simply the right tool. For watching residual streams with hooks on layer 22 — it never will be, and that's fine. Different instruments, different music.SGLang-Omni 위의 Higgs가 실제로 주는 것 — 그리고 얻는 법. 지원되는 하드웨어(빌린 CUDA 박스)에서 sgl-omni serve --model-path bosonai/higgs-tts-3-4b(저장소가 higgs-audio-v3-tts-4b에서 리네임됐고 옛 이름은 리다이렉트된다)를 치면 위의 공개 H100 수치를 얻는다 — 하지만 TTS에서 더 깊은 이득은 스트리밍이다: 첫 코덱 프레임 직후의 첫 PCM 청크(서버 기본값은 첫 오디오 청크를 프레임 1개로 잡는다) 대 우리 포트의 전부-생성-후-보코딩 12.7초. 이 프로젝트의 워크로드와 맞아떨어지는 지점도 있다: SGLang-Omni의 프리픽스 캐싱(RadixAttention)은 공유 프롬프트 프리픽스의 KV 캐시를 재사용하는데, 우리 16방향 스윕은 소스 언어마다 레퍼런스 목소리 하나를 재사용한다 — 소스별 ~0.65초의 프리필을 네 번이 아니라 한 번만 내면 되고, 스티어링 α-스윕(모든 α에 고정 레퍼런스 하나)은 정확히 한 번만 내면 된다. 대량 생성에는 서빙 스택이 그냥 맞는 도구다. 22번 레이어에 훅을 걸고 residual stream을 지켜보는 일에는 — 영원히 아닐 것이고, 그래도 괜찮다. 악기가 다르면 연주도 다르다.
Beyond inference추론 너머

The same laptop ran the whole research pipeline같은 노트북이 연구 파이프라인 전체를 돌렸다

Synthesis was the easy half. The interpretability study also attached read-only forward hooks to all 36 backbone layers during generation and captured a residual-stream vector per layer per clip — hundreds of captures across languages, genders, accents, and emotions. Three habits kept that inside 32 GB: hook sparsely (mean-pool per layer instead of storing [seq × hidden]), move every captured tensor to CPU immediately, and del + torch.mps.empty_cache() between runs. Budget roughly the driver's 13 GB for the model and keep captures out of GPU memory entirely — then a 4B audio LLM is a perfectly workable interpretability substrate on a laptop.합성은 쉬운 절반이었다. 해석가능성 연구는 생성 도중에 36개 백본 레이어 전부에 읽기 전용 forward hook을 걸고 클립마다 레이어별 residual-stream 벡터를 캡처했다 — 언어·성별·억양·감정에 걸쳐 수백 번의 캡처다. 그걸 32 GB 안에 가둔 세 가지 습관: 희소하게 후킹하기([seq × hidden]을 쌓지 말고 레이어별 mean-pool), 캡처한 텐서는 즉시 CPU로 옮기기, 런 사이마다 del + torch.mps.empty_cache(). 모델 몫으로 드라이버의 13 GB쯤을 잡아두고 캡처는 아예 GPU 메모리 밖에 두면 — 4B 오디오 LLM도 노트북 위에서 충분히 작업 가능한 해석가능성 기판이 된다.
