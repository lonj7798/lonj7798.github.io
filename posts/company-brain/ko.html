기업들은 실제로 어떻게 굴리나
검색·갱신 여정 해부

실제로 돌아가는 물건들을 뜯는다. Glean, Microsoft 365 Copilot, Atlassian Rovo, Slack AI 같은 상용 플랫폼과 Uber·LinkedIn·DoorDash의 사내 구축 사례까지 — 세 겹으로 판다: user message가 프롬프트가 되기까지의 전 구간, 새 정보가 들어왔을 때의 세 갈래 갱신 결정, 그리고 AI를 도메인 전문가로 만드는 수직 도메인 플레이북. 각 카드의 검색 여정은 실제 순서대로 점등된다.

01

두 진영 — 사서 붙이거나, 직접 짓거나

기업 사례를 늘어놓고 보면 진영이 둘로 갈린다. 어느 쪽이든 밑바닥 구조는 같은 5단계 파이프라인(수집→구조화→검색→합성→검증)이다. 갈리는 건 "누가 그 파이프라인을 소유하느냐"다.

플랫폼 진영 — 연결하면 끝

  • Glean — 커넥터 100+짜리 독립 검색 플랫폼. "컴퍼니 브레인" 상용화의 원형.
  • Microsoft 365 Copilot — 테넌트마다 자동으로 시맨틱 인덱스를 깔아주는, 배포 규모로는 최대의 사례.
  • Atlassian Rovo — Jira·Confluence를 중심에 두고 1,500억 객체·관계 규모의 Teamwork Graph를 세움.
  • Slack AI — 새 인덱스를 만들지 않고 Slack의 기존 검색 스택을 RAG의 리트리버로 재활용.

공통 승부처: 커넥터 폭과 권한 동기화. 검색 품질만큼이나 "누가 뭘 봐도 되는가"를 파는 장사다.

사내 구축 진영 — 좁게 파서 깊게

  • Cerebras Knowledge — 단일 Postgres 테이블 위의 사내 생태계. 하루 15,000+ 질문을 받는다.
  • Uber Genie — 월 4만 5천 건의 질문이 올라오는 Slack 지원 채널들을 지키는 온콜 코파일럿.
  • LinkedIn 고객지원 KG-RAG — 과거 티켓을 지식그래프로 바꿔 상담 해결 시간을 28.6% 줄임.
  • DoorDash Dasher 봇 — 가드레일과 심판(LLM Judge)을 붙여 환각을 90% 줄인 지원 챗봇.

공통 승부처: 도메인 하나를 정해 정확도를 사람 수준까지 끌어올리는 것. 범용성은 처음부터 버린다.

덧: Notion도 이 판에 있다. 1억 사용자 대상 엔터프라이즈 검색의 추론을 Cerebras 하드웨어에서 돌려 300ms 안에 결과를 스트리밍한다 — 지식베이스 경쟁이 검색 알고리즘을 넘어 추론 지연(latency) 싸움까지 왔다는 신호다.
02

검색 여정 비교기 — 눌러서 나란히 보기

이름을 누르면 그 시스템에서 질문 하나가 답이 되기까지의 단계가 순서대로 나타난다. 오픈소스 레퍼런스(Onyx·GraphRAG·LightRAG·Karpathy 위키)도 넣어뒀으니 상용·사내·오픈소스를 한 자리에서 견줄 수 있다.

03

질문의 해부 — user message가 프롬프트가 되기까지

앞의 비교기가 "어느 시스템이 어떤 여정을 도는가"를 그렸다면, 이 장은 여정의 한 칸 한 칸을 연다. 사용자가 친 문장 하나가 어떻게 가공되고, 어디에 어떤 형태로 꽂히는지 — 공개된 파이프라인 문서와 실제 프롬프트 원문으로 재구성했다.

먼저 결론 하나. 조사한 시스템 전부에서 사용자의 원문이 그대로 검색 엔진에 들어가는 경우는 없다. Azure의 intent 프롬프트, LangChain의 condense 프롬프트, Uber의 Query Optimizer, DoorDash의 대화 요약, Cerebras의 플래너, LinkedIn의 개체·의도 추출 — 채팅창과 인덱스 사이엔 반드시 LLM 호출이 최소 한 번 낀다. 그런데 반대편 끝에선 정반대다. 생성 모델에게는 거의 항상 원문이 간다. 재작성본은 검색을 조준하기 위한 도구이고, 원문은 뉘앙스를 보존한 채 답변 생성에 쓰인다. 이 분업(검색은 재작성본, 생성은 원문)이 시스템들을 관통하는 가장 일관된 패턴이다.

질의 변환 추적기 — 예시를 골라 단계별 변형을 따라가기

* 각 시스템의 공개 문서에 기술된 메커니즘(Azure intent 생성, M365 인접어 확장, E5 접두사 계약, Azure AI Search ACL 필터 문법 등)을 바탕으로 재구성한 예시다. 실제 내부 문자열은 회사마다 다르다.

0단계 — 관문: 재작성보다 분류기가 먼저다

원문이 LLM 재작성기에 닿기도 전에, 값싼 분류기 여러 개가 병렬로 지나간다. 전부 "다시 쓰기"가 아니라 "판정"이다 — 공격인가, 개인정보가 섞였는가, 애초에 답할 질문인가.

프롬프트 공격 분류
M365 Copilot은 jailbreak(UPIA) 분류기와 XPIA(cross-prompt injection attack) 분류기를 모델 실행 전에 돌려 고위험 프롬프트를 차단한다. 같은 기능의 부품도 판다: Azure Prompt Shields(사용자 프롬프트/문서 이중 검사), AWS Bedrock의 PROMPT_ATTACK 필터, Google Model Armor, Meta Prompt Guard 2(22M 파라미터짜리 초소형 분류기). Glean은 자사 게이트(Glean Protect)의 수치를 공개했다: 직접 주입 탐지 97.8%, 간접 주입 90%.
신뢰 경계 태깅 — 디테일 하나
Bedrock은 사용자 입력을 <amazon-bedrock-guardrails-guardContent_xyz> 태그로 감싸고, 접미사 xyz를 요청마다 랜덤 생성하라고 권한다. 공격 문장은 시스템 지시문과 구분이 안 되므로, "어디까지가 사용자 입력인가"를 예측 불가능한 태그로 표시하는 것. 태그가 없으면 공격 필터가 아예 안 돈다고 문서에 명시돼 있다.
PII 마스킹
검색·로그로 흘러가기 전에 개인정보를 치환한다. Bedrock 민감정보 필터는 31종 엔티티를 {NAME}·{EMAIL} 식으로 마스킹(입력은 마스킹, 출력은 차단 — 방향별로 다른 정책 가능). 오픈소스 기본값은 Microsoft Presidio. 단, Bedrock 문서의 경고가 뼈아프다: CloudWatch 호출 로그에는 마스킹 전 원문이 그대로 남는다.
질문 유형 분류 → 정중한 거절
Vertex AI Search의 answer API는 질의를 ADVERSARIAL_QUERY, NON_ANSWER_SEEKING_QUERY(잡담), JAIL_BREAKING_QUERY 등으로 분류하고, 걸리면 생성을 거부한 채 고정 문구 ("A summary could not be generated…")와 검색 결과만 돌려준다. 차단이 아니라 기능 강등이 기본 동작이라는 게 포인트.
관문은 확률적이다. 2025년 공개된 EchoLeak(CVE-2025-32711)은 조작된 메일 한 통으로 M365 Copilot의 XPIA 분류기를 우회해 무클릭 데이터 유출까지 갔다. Microsoft 자신도 문서에 "추가 검증 계층을 항상 두라"고 쓴다. 그리고 한국어 사용자라면 하나 더: Prompt Shields의 학습 언어 목록에 한국어가 없다(Model Armor에는 있다). 한국어 컴퍼니 브레인의 0단계는 기성품 검증부터 직접 해야 한다.

1단계 — 재작성: 대화를 독립 질의로

멀티턴 대화의 마지막 메시지는 그 자체로 검색이 안 된다("그건 어떻게 됐어?"). 그래서 모든 시스템의 첫 LLM 호출은 대화 이력 + 마지막 메시지 → 독립 질의 변환이다. RAG 역사상 가장 많이 복제된 프롬프트가 바로 이것이다: 

# LangChain CONDENSE_QUESTION_PROMPT — 원문 그대로
Given the following conversation and a follow up question, rephrase the
follow up question to be a standalone question, in its original language.

Chat History:
{chat_history}
Follow Up Input: {question}
Standalone question:
Azure — 재작성이 별도 과금 항목이다
Azure OpenAI "On Your Data"는 이 단계를 intent 프롬프트라는 별도 모델 호출로 돌린다. 결과물은 질의 하나가 아니라 검색 의도의 목록(재작성과 분해를 한 번에). 측정된 비용까지 공개돼 있다: intent 프롬프트 811~1,385 토큰, 출력 18~27 토큰. 의도별로 각각 검색이 나간다.
DoorDash — 재작성이 아니라 요약
배달 기사의 문제는 메시지 여러 개에 흩어져 온다. 그래서 "마지막 메시지 재작성" 대신 대화 전체 → 핵심 이슈 한 문장으로 압축하고, 그 요약이 검색 키가 된다. KB 아티클도 [이슈|해법] 구조로 쪼개 이슈 부분만 임베딩해 두므로, 결과적으로 이슈-대-이슈 매칭이 된다.
Uber — 코퍼스 카탈로그를 재작성 프롬프트에 넣는다
EAg-RAG의 Query Optimizer는 모호한 질의를 다듬고 복잡한 질의를 쪼개는데, 이때 문서 목록 아티팩트(제목·요약·FAQ)가 프롬프트 컨텍스트로 들어간다. "우리가 뭘 갖고 있는지"를 알고 재작성하는 것. 이어지는 Source Identifier가 답이 있을 문서군을 좁힌다.
한도들
Vertex AI Search의 자동 재작성(maxRephraseSteps)은 최대 5단계. Azure는 질문+이력을 합쳐 2,000 토큰에서 자른다. Cerebras는 플래너가 "무엇이 인덱스에 있는지의 압축 기술서"를 보고 도구를 고른다 — 재작성과 라우팅이 한 호출에 합쳐진 형태다.

2단계 — 확장과 분해: 질의가 여러 개가 된다

독립 질의 하나로도 부족하다. 임베딩 유사도는 어휘가 어긋나면 놓치고, 멀티홉 질문은 한 번의 검색으로 안 풀린다.

확장 — 같은 뜻, 다른 말

  • M365 시맨틱 인덱스 — "디자인을 칭찬한(praised) 메일"을 찾으면 인접어 elated·excited·amazed까지 검색 범위에 포함. 어형 변화(tech/technology/technologies)와 동의어도 흡수.
  • Glean — 확장을 기업어 문제로 취급: 약어·프로젝트명·사내 은어 처리. 로그로 남기는 재작성 산출물 네 가지를 문서에 명시한다: 철자 교정 · 동의어 확장 · 질의 확장 · 의도 라우팅.
  • LangChain MultiQueryRetriever — "이 질문의 서로 다른 버전 3개를 생성하라"는 프롬프트로 패러프레이즈 확장. 거리 기반 유사도 검색의 한계를 질의 다양화로 우회.
  • HyDE — 질문으로 가짜 답 문서를 생성해 그걸 임베딩. 어려운 질문→문서 매칭을 쉬운 문서→문서 유사도로 바꾼다.

분해 — 한 질문을 여러 검색으로

  • Bedrock Knowledge Bases — QUERY_DECOMPOSITION 플래그 하나로 켠다. 문서의 예시: "2022 월드컵에서 아르헨티나와 프랑스 중 누가 더 많이 득점했나" → "아르헨티나 득점은?" + "프랑스 득점은?" 두 개의 하위 질의.
  • Azure — intent 목록 자체가 분해다. 의도마다 별도 검색 → 청크 수집 → 병합.
  • Vertex — "Andie Ram과 Arnaud Clément의 공통 직업·취미는?"을 4개 하위 질의로 쪼개는 예시를 문서에 실어 놓았다.
  • LinkedIn KG-RAG — 분해가 아니라 추출: LLM이 개체명(Map 형태)과 의도 집합을 뽑고, 그걸 그래프 진입점 매칭에 쓴다. 최종적으로 질의는 Cypher 쿼리로까지 번역된다.

3단계 — 질의의 세 가지 몸

재작성·확장을 마친 질의는 리트리버로 나가는 순간 세 가지 물리적 형태로 갈라진다. 같은 문장이 임베딩 문자열, 어휘 검색식, 필터 표현식이 되어 병렬로 발사되고, 결과는 나중에 융합(RRF 등)된다.

① 임베딩 문자열 — 접두사는 인코더와의 계약
E5 계열은 질문에 query:, 문서에 passage: 접두사가 필수다. 제작자 FAQ 원문: "그렇게 학습됐다. 안 붙이면 성능 저하를 보게 될 것." BGE는 질문 쪽에만 지시문("Represent this sentence for searching relevant passages:")을 붙이고, Cohere는 접두사 대신 API 파라미터(input_type=search_query/search_document)로, OpenAI text-embedding-3는 아무것도 안 붙인다. 접두사 여부는 RAG 프레임워크가 아니라 인코더의 속성이다.
② 어휘 검색식 — 필드 부스트
Elasticsearch식 정석: "fields": ["subject^3", "message"] — 제목 히트가 본문 히트의 3배 점수. Glean은 여기에 신선도 가중치·권위 점수·인기 신호를 얹고, 의도 코호트별로 다른 필드를 민다 (티켓 ID는 정확 일치 필드, 정책 질문은 canonical 소스, 하우투는 최신 절차 문서). Cerebras의 근거는 명쾌하다: "엔지니어가 에러 메시지를 그대로 붙여넣으면, 정확한 어휘 일치가 거의 항상 최선의 증거다. 어떤 의미 유사도도 그걸 이겨선 안 된다."
③ 필터 표현식 — 권한·메타데이터·시간
ACL은 검색 후 거르기가 아니라 질의에 붙는 필터다. Azure AI Search의 정석 문법: group_ids/any(g:search.in(g, 'g1,g2')) — 질문자의 그룹 목록이 모든 질의에 부착된다. M365는 grounding 단계 자체가 "질문자가 권한 가진 콘텐츠만" 접근하고, Slack AI는 "사용자가 이미 볼 수 있는 데이터에만" 작동한다. Cerebras의 프로젝트 스코프, M365의 SharePoint 라이브러리 스코프처럼 범위 지정도 필터의 일종이다.
병렬 fan-out
Cerebras: 플래너가 고른 도구들을 executor가 병렬 실행 → 공통 증거 스키마로 정규화. Anthropic의 컨텍스트 검색 레퍼런스 아키텍처도 질의당 BM25 + 임베딩 두 리트리버를 돌려 랭크 융합한다. M365 오케스트레이터는 후보 스킬 5개 슬롯을 어휘 일치→의미 일치 순서로 채운다. Azure는 의도별로 검색을 낸다. 단일 검색 호출로 끝나는 시스템은 이 판에 없다.

4단계 — 프롬프트 조립: 최종 프롬프트의 지리학

왜 질문이 맨 끝인가 — 2023년의 근거
Liu et al.의 Lost in the Middle(2023): 관련 정보가 컨텍스트 중간에 있을 때 성능이 U자로 꺼진다. GPT-3.5-Turbo, 문서 20개 기준 — 정답 문서가 1번 위치면 75.8%, 중간(10번)이면 53.8%, 끝(20번)이면 63.2%. 중간 배치는 문서를 아예 안 준 경우(56.1%)보다도 나빴다. Anthropic 문서의 처방("긴 자료는 위, 질의는 끝 — 최대 30% 개선")은 2026년 현재도 Claude 5 세대 문서에 그대로 살아 있고, Google도 같은 조언, OpenAI는 "지시문을 앞뒤 양쪽에" 변형을 권한다.
2026년의 갱신 — 문제는 '중간'이 아니라 '길이'
최신 모델들은 단순 중간 위치 편향은 대부분 극복했다(LongPiBench 2025: "most current models are robust against the lost-in-the-middle issue"). 대신 길이와 간섭이 살아있는 문제다: NoLiMa(ICML 2025)는 어휘 겹침을 제거하면 128K 이상을 표방하는 모델들의 유효 컨텍스트가 1~8K 수준으로 쪼그라든다고 측정했고, Gemini 3 Pro 모델 카드조차 8-needle 1M 토큰에서 26.3%를 자인한다. Chroma의 Context Rot 연구(모델 18개): 사소한 과제조차 입력이 길어지면 성능이 떨어진다. 결론 — 리랭커로 top-10만 남기고 압축하는 파이프라인은 이제 "중간 회피"가 아니라 컨텍스트 위생으로 정당화된다.
토큰 예산 — 공개된 유일한 산식 (Azure)
질문+이력 ≤ 2,000 토큰 · 응답 몫 20% 예약 · 시스템 메시지는 모델별 상한(400~4,000 토큰, 내부 베이스 프롬프트에 덧붙는 구조라 초과분은 잘림) · 나머지 전부가 청크 몫 (기본 5개 × 1,024 토큰). 실측 프롬프트 크기 3,997~4,854 토큰. 한편 Anthropic의 컨텍스트 검색 실험은 top-20 청크가 5·10개보다 좋았다고 보고한다 — 예산과 품질의 트레이드오프는 아직 합의가 없다.
인용 — 예절에서 구조로
3세대가 공존한다. ① 프롬프트 지시: "모든 주장에 인용을 달라"(Azure, 인용 안 된 문서는 UNCITED_REFERENCE로 표시되어 돌아옴). ② API 구조: Anthropic Citations API — 문서를 청크 단위로 넘기면 유효성이 보장된 인용 포인터가 응답에 붙고, 인용문(cited_text)은 출력 토큰 과금에서 빠진다. ③ 사후 검증: DoorDash의 2단 가드레일 — 1차는 응답-근거 간 의미 유사도(싼 검사), 걸리면 2차 LLM 평가로 근거성·컴플라이언스 판정.
더 깊이: 청크 포맷팅과 컨텍스트 순서의 실무 — XML 태그 · 상황 헤더 · LongContextReorder

청크 포맷. Anthropic 공식 권고는 문서마다 <document index="n"> 안에 <source>와 <document_content>를 넣는 XML 구조다. index 속성이 그대로 인용 ID가 된다. 지시문·컨텍스트·입력이 섞일 때 XML 태그가 경계를 명확히 한다는 게 근거. Cerebras도 스레드 정규화 포맷의 참고문헌으로 이 문서를 인용한다.

상황 헤더(contextual retrieval). 임베딩 전에 청크마다 LLM이 "이 청크는 전체 문서에서 어떤 위치인가" 한두 문장을 생성해 붙인다. Anthropic 측정으로 검색 실패율이 5.7%→3.7%(-35%), BM25 결합 시 2.9%(-49%), 리랭킹까지 얹으면 1.9%(-67%).

순서. 관련도 내림차순이 기본이지만, LangChain은 LongContextReorder라는 변환기를 실제로 배포했다 — 소스 독스트링 원문: "the less relevant documents will be at the middle of the list and more relevant elements at beginning / end." 알고리즘은 4줄이다: 관련도 정렬 목록을 뒤집고, 홀수 번째는 꼬리에, 짝수 번째는 머리에 번갈아 꽂는다 (1등이 한쪽 끝, 2등이 반대쪽 끝, 꼴찌들이 가운데). 2025년 벤치마크 기준 절대 위치 편향이 줄어든 최신 모델에선 기대 이득이 작아진, "2023년식 완화책이지만 여전히 출하 중"인 부품이다.

이 장의 한 줄
질의는 검색용 사본(재작성·확장·접두사·필터)과 생성용 원문으로 분리된다. Cerebras는 리랭커와 합성 LLM에 "원래 질문"을 넘긴다고 명시하고, Azure는 청크 옆에 "사용자 질문과 대화 이력"을 그대로 싣는다. 검색은 기계가 읽기 좋은 질의를, 생성은 사람이 쓴 그대로의 질의를 원한다 — 한 문장이 두 개의 삶을 사는 것이 현대 RAG의 기본 설계다.

처음 세우는 법

3단 사다리로 따라간다: 웹훅은 5분 내 처리, 증분 크롤은 커넥터별 10분~6시간 (Salesforce 10분 · Slack/Drive 3시간 · Jira 6시간), 전체 크롤은 대개 28일 주기. 권한 변경도 같이 동기화된다. 랭킹 모델은 self-tuning이라 사용 기록이 쌓일수록 그 회사에 맞게 좋아진다 — 자체 발표로는 6개월 연속 학습에 검색 품질 약 20% 개선.
권한 처리의 급이 다른 점
인덱스 시점 스냅샷도, 생성 후 필터도 아니고 질의 시점에 매번 ACL을 강제한다. 규제 산업에서 배포 가능한 이유가 이 한 줄이다.

"엔지니어는 Salesforce 문서를 찾는 게 아닐 것"이라는 식의 개인화가 검색 품질의 절반이다. 쿼리와 문서만 보는 오픈소스 RAG들이 놓치는 축 — 질문한 사람이 누구인가 — 를 랭킹의 1급 신호로 만든 게 Glean의 해자다.

갱신의 세 갈래 — 새 정보가 들어올 때

검색 여정이 읽기 경로라면, 이 장은 쓰기 경로다. 새 정보가 도착하면 시스템은 기존 지식과 비교해 세 갈래 중 하나로 보낸다: 겹치거나, 안 겹치거나, 비슷하지만 다르거나. 갈래마다 실재하는 메커니즘이 있고, 이름과 임계값까지 공개돼 있다.

분기 전에 0번 관문이 있다. LLM을 부르기 전에 싸게 끝내는 동일성 검사들: Cerebras는 Slack 이벤트의 고유 이벤트 ID로 중복 수신을 버리고, LlamaIndex 인제스트 파이프라인은 doc_id + 콘텐츠 해시 맵으로 "같은 문서, 안 바뀜"을 건너뛰고, 구글 크롤은 64비트 SimHash에 해밍 거리 ≤ 3이면 근사 중복으로 처리하며(수십억 페이지 규모에서 검증된 파라미터), 학습데이터 중복 제거의 표준인 SemDeDup은 코사인 유사도 ≥ 0.99(ε=0.01, NVIDIA NeMo 기본값)를 의미 중복으로 본다. 이 관문을 통과한 — 즉 "확실한 중복은 아닌" — 후보만 아래 결정 트리로 들어간다.

판정기의 실물 — Mem0의 두 단계

"비교해서 세 갈래로 보낸다"를 가장 구체적으로 공개한 건 메모리 시스템 Mem0다(2025 논문 + 오픈소스). 추출 단계에서 LLM이 대화에서 후보 사실들을 뽑고, 갱신 단계에서 후보마다 벡터 검색으로 유사 메모리 top-10을 가져와 LLM에게 함수 호출 인터페이스로 들이민다 — 선택지는 ADD / UPDATE / DELETE / NOOP 네 개. 이 장의 세 갈래가 그대로 도구 네 개가 된 모양새다. 효과로 내세우는 수치는 풀컨텍스트 대비 p95 지연 91% 감소(17.1초 → 1.44초), 토큰 비용 90% 이상 절감 — 단, 경쟁 시스템 비교 수치는 Mem0 자체 논문의 것이니 그만큼만 믿자.

규모의 문제 — 커넥터 동기화 사다리

문서 단위 시스템에서 세 갈래는 커넥터 동기화로 구현된다. 패턴은 어디나 같은 3단 사다리다: 웹훅(분 단위) → 증분 크롤(분~시간) → 전체 크롤(주 단위, 최후의 정합성 보장). 아래 수치는 각 사가 문서에 공개한 기본값이며, 배포별 보장이 아니다.

충돌이 질의 시점까지 살아남으면 — 랭킹이 판정한다

쓰기 경로에서 못 거른 모순(옛 정책 문서 vs 새 정책 문서)은 결국 랭킹이 심판한다. Glean의 신호 스택: 신선도 가중치 + 권위 점수 + 사용 신호. 그들이 문서에 적어둔 균형 감각이 좋다 — "과하게 튜닝된 최신성 부스트는 새롭지만 엉뚱한 문서를 canonical 정책 페이지 위로 올린다", 그리고 "몇 년째 수정 안 된 WiFi 비밀번호 문서도 사람들이 계속 찾으면 여전히 권위 있다." Cerebras는 같은 문제를 RRF에 age decay를 섞어 "동점이면 새 스레드가 이긴다"로 푼다. 자동 판정이 안 되는 낡음은 사람에게 돌려보낸다: Glean Verification은 문서 소유자에게 재검증 리마인더를 걸고, 낡은 문서엔 Deprecated 배지를 붙여 지우지 않고 경고한 채 남긴다.

프로덕션의 실물 — DoorDash가 해결된 상담을 지식으로 만드는 법

"새 정보가 지식베이스에 편입되는" 전 과정을 회사 규모로 공개한 사례는 DoorDash가 유일하다(2025). 상담이 인간 상담원으로 에스컬레이션됐다는 것 자체를 봇의 실패 신호로 삼아, 그 대화들을 지식 후보로 만든다.

파이프라인 8단계
익명화된 상담 중 에스컬레이션된 것만 선별 → 임베딩 후 온라인 군집화(신규 대화를 기존 클러스터 중심점과 코사인 비교, 임계값 0.70~0.90 미달이면 새 클러스터 생성) → 상위 클러스터 인간 검수 (중복 클러스터 수동 병합) → 클러스터를 "실행형 문제 vs 정보형 질문"으로 분류 → 정보형에 대해 LLM이 이슈 요약 + 모범 상담 해결례로 아티클 초안 작성 → 콘텐츠 전문가 검수(정책 정확성·톤·엣지 케이스) → 승인분 임베딩·배포 → 기존 RAG로 서빙.
성과 — 조건 붙여 읽기
고트래픽 클러스터의 에스컬레이션율이 대조군 78% → 처리군 43%(A/B 실험). 처리군 KB 검색 이벤트의 약 75%가 이 파이프라인이 만든(UGC 유래) 콘텐츠만 반환. 전역 수치가 아니라 고트래픽 클러스터 한정, 기간·볼륨 비공개 — 회사 블로그 수치의 정석적 한계. 덧: DoorDash엔 별개의 −90%가 하나 더 있다 — 2026년 시뮬레이션 플라이휠이 배포 전 환각률을 약 90% 줄인 수치로, 2024년 가드레일의 프로덕션 −90%와 다른 값이다. 혼동 금지.
세 갈래와의 대응
분기 B(새 지식 발견·초안)는 기계가 한다. 분기 A(겹침)는 클러스터 임계값 + 수동 병합. 그런데 분기 C(기존 아티클과의 모순)는 자동화가 없다 — 생성된 아티클과 기존 KB의 충돌 검사 메커니즘이 공개 자료 어디에도 없고, 인간 검수가 그 역할을 겸한다.
메모리 시스템과의 대비
Mem0·Graphiti·Letta는 세 갈래 전부를 자동화한다 — 단, 사용자 한 명의 메모리 규모에서. DoorDash는 탐지와 초안까지 자동화하고 쓰기는 인간 뒤에 세운다 — 회사 전체가 보는 지식이라서. 변수는 기술이 아니라 잘못 쓴 지식의 폭발 반경이다.
이 장의 한 줄
갱신의 실무는 "지식을 고친다"가 아니라 분기를 설계한다에 가깝다. 겹치면 버리고(그것도 명시적 연산으로), 새로우면 전역 재계산 없이 붙이고, 비슷하면 세 유파(덮어쓰기 / LLM 판정 / 시간적 무효화) 중 데이터의 성격에 맞는 걸 고른다. 그리고 무엇을 자동화하고 무엇을 인간 뒤에 세울지는 정확도가 아니라 폭발 반경으로 정한다.

수직 도메인 — AI를 도메인 전문가로 만드는 법

컴퍼니 브레인의 플레이북(수집 → 구조화 → 검색 → 합성 → 검증)을 한 직업 영역에 통째로 밀어넣으면 수직 도메인 AI가 된다. 법률의 Harvey, 금융의 Hebbia, 의료의 OpenEvidence·Abridge — 그리고 사실 이 리포트의 사내 구축 사례들도 전부 수직이다. Cerebras KB의 수직은 반도체 엔지니어링이고, DoorDash의 수직은 배달 물류 지원이다.

사례들을 비교 가능하게 만들기 위해 일곱 개의 레버로 분해했다: ① 코퍼스(전문가들이 이미 신뢰하는 도메인 데이터를 확보했는가 — 라이선스든 자체 축적이든), ② 구조(도메인의 온톨로지·스키마를 데이터에 입혔는가), ③ 워크플로(전문가의 작업 분해를 제품에 새겼는가), ④ 평가(전문가가 만든 골든셋·벤치마크로 재는가), ⑤ 피드백 루프(전문가의 교정이 시스템으로 되돌아가는가), ⑥ 모델 전략(파인튜닝인가 검색인가, 또는 둘 다인가), ⑦ 인터페이스(전문가가 이미 사는 도구 안으로 들어갔는가). 탭을 눌러 사례별 프로필을 보자 — 레버 바는 공개 자료 기반의 정성 평가다.

파인튜닝이냐 검색이냐 — 업계가 이미 치른 수업료

반면교사 — BloombergGPT (2023)
3,630억 토큰의 자체 금융 코퍼스로 50B 모델을 처음부터 학습($10M 이상 추정). 결과는 유명하다 — 범용 GPT-4가 대부분의 금융 태스크에서 이겼다. Bloomberg가 실제로 출시한 건 그 모델이 아니라 터미널 안의 검색 기반 제품이었다: 2억 건+ 문서 위의 Document Insights, 원문 발췌로 점프하는 transparency link. "자체 도메인 GPT를 학습한다"는 전략은 "프런티어 모델을 자기 코퍼스에 접지(ground)시킨다"에 패배했다.
예외 조항 — 채점 가능한 규칙서가 있을 때
Ambience Healthcare는 OpenAI RFT(강화 파인튜닝)로 ICD-10 의료코딩 모델을 만들어 보드 인증 의사 18명 대비 상대 27% 우위를 얻었다(4인 이상 전문의 합의로 라벨링한 골든 패널 평가). 왜 여기선 통했나 — ICD-10은 검증 가능한 규칙서와 채점 가능한 정답이 있는 좁은 과제다. Harvey의 판례 모델(10B 토큰 상당 mid/post-training, 변호사 97%가 GPT-4 대비 선호 — OpenAI 자체 발표)도 같은 패턴. "열린 이해"는 검색으로, "닫힌 채점"은 파인튜닝으로.

수직 vs 수평 — 양쪽의 논리

수직이 이긴다는 논리 (강세론)

  • 시장의 크기가 다르다 — 수직 AI는 IT 예산이 아니라 인건비·서비스 예산을 공략한다(Bessemer: "전통 소프트웨어 시장의 10배 TAM").
  • 평가 기준을 소유한다 — 중요한 벤치마크가 전부 수직 회사에서 나왔다. Harvey의 BigLaw Bench와 LAB(과제 1,200개+, 전문가 작성 루브릭 기준 75,000개+, 하나라도 틀리면 실패인 all-pass 채점), Sierra의 τ-bench(정책 준수 고객응대에서 GPT-4o급 에이전트 성공률 50% 미만, 8회 재시도 신뢰도 pass^8은 소매 도메인 기준 25% 미만).
  • 책임을 지는 장치 — OpenEvidence는 근거가 불충분하면 답을 거부하고, Abridge는 노트의 모든 문장을 원본 오디오로 역추적(Linked Evidence), Sierra는 해결된 상담에만 과금한다. 프런티어 모델이 기본 탑재하지 않는 층이다.
  • 데이터 루프 — EvenUp의 축적된 합의금·판결 DB, Abridge의 주간 모델 업데이트(의사 교정 반영), Sierra의 실패 마이닝. 라이선스로 살 수 없는 결과 데이터.

수평이 흡수한다는 논리 (약세론)

  • 기반 모델이 지식층을 흡수 중 — Harvey 자신의 데이터가 증거다: BigLaw Bench에서 베이스 모델 점수가 두 세대 만에 ~60%(GPT-4o/Claude 3.5) → ~90%(GPT-5/Claude 4.5/Gemini 3).
  • 연구소의 직접 진출 — OpenAI의 의료 3연타: ChatGPT Health(2026-01-07, 소비자) → ChatGPT for Healthcare(01-08, 병원 엔터프라이즈) → ChatGPT for Clinicians(04-23, NPI 인증 의사 무료) — OpenEvidence의 $12B 라운드 석 달 뒤에 같은 지점을 정조준했다. HealthBench Professional에선 ChatGPT for Clinicians(GPT-5.4 기반)가 59.0점, GPT-5.6 Sol이 60.5점으로 의사 응답(43.7점, 시간 무제한+웹 검색 허용 조건)을 앞선다 — 단, OpenAI가 만들고 OpenAI가 채점한 벤치마크라는 각주는 항상 붙는다.
  • 수평 플랫폼의 수직 팩 — Glean은 에이전트 라이브러리로 영업·엔지니어링·IT·HR용 프리빌트 에이전트 30개+를 얹었다. 인터페이스를 쥔 기존 강자도 있다 — Epic이 자체 "Art for Clinicians"를 내며 Abridge의 앞마당에 들어왔다.
종합하면 — 지식(암기된 도메인 상식)은 기반 모델이 흡수하고 있고, 그 속도는 수직 회사들의 자체 벤치마크가 증명한다. 수직이 계속 이기는 곳은 다섯 가지다: 닫힌 우주의 작업 결과물(all-pass로 채점되는 실제 업무 산출물), 결과 데이터 루프, 책임·감사 장치, 전문가의 도구 안(Word·Epic·터미널·Slack), 결과 기반 과금. 그리고 두 진영은 싸우면서 서로 스며든다 — Harvey는 M365 Copilot 안의 에이전트로 들어갔고, Sierra 에이전트는 ChatGPT에서 뜬다.

내 도메인에 적용한다면 — 사례들이 합의하는 순서

1–5: 기반 다지기

  • 1. 과금 단위의 일을 목록화하라, 주제 말고. Harvey는 BigLaw Bench를 변호사의 실제 청구 시간 기록(time entries)에서 만들었다. 전문가가 돈 받는(또는 시달리는) 과제 10~20개를 "지시문 + 닫힌 문서 우주 + 기대 산출물"로 적는 게 시작이다.
  • 2. 전문가가 이미 신뢰하는 코퍼스를 확보하라. 라이선스하거나(OpenEvidence×NEJM·JAMA) 스스로 축적하라(Abridge의 동의받은 150만 진료 대화). 어느 쪽이든 스크래핑은 아니다.
  • 3. 임베딩 전에 도메인 구조를 입혀라. SOAP 노트, ICD-10, 규제 결정 트리(Norm Ai), 티켓 트리(LinkedIn), Cerebras의 {질문/요약/해결/시스템/코드참조} 스키마 — 구조가 검색을 전문성으로 바꾼다.
  • 4. 전문가의 작업 분해를 워크플로로 새겨라. 메가 프롬프트가 아니라 Hebbia의 ISD 스텝, Sierra의 Journeys, EvenUp의 조립 라인처럼.
  • 5. 확장 전에 SME 골든셋부터, 채점은 all-pass로. Uber의 v1이 살아난 건 퇴짜 놓을 골든셋이 있었기 때문이다.

6–10: 확장하며 지키기

  • 6. 기본은 검색+프런티어 모델. 파인튜닝은 채점 가능한 규칙서가 있는 곳만. BloombergGPT가 수업료를 대신 냈고, Ambience RFT가 예외의 조건을 보여줬다. 서브태스크별 멀티모델 라우팅(Harvey의 Auto)도 표준이 됐다.
  • 7. 전문가의 기존 화면 안으로 들어가라. Word 애드인, Epic의 첫 "Pal", 터미널 패널, Slack 봇, Claude Code로 들어가는 MCP. 새 목적지 앱은 마지막 수단.
  • 8. 모든 답을 감사 가능하게. Linked Evidence(노트→오디오), transparency link(요약→원문 발췌), 인용(주장→논문), source score.
  • 9. 전문가 교정을 정해진 주기로 되먹여라. Abridge는 의사 편집을 주간 업데이트로, EvenUp은 사내 전문가 100+명의 검수를 모델 개선으로.
  • 10. 거절 경로를 답변 경로만큼 공들여 설계하라. OpenEvidence는 근거 불충분이면 무응답, DoorDash는 가드레일 탈락 시 인간 연결, Sierra는 실패 건 무과금. 전문가 도메인에서 안전한 무응답은 신뢰 그 자체다.
이 장의 한 줄
"AI를 도메인 전문가로 만든다"의 실체는 모델 교체가 아니라 일곱 레버의 동시 조작이다. 그중 기반 모델이 대신해 주는 건 여섯 번째(모델) 하나뿐이고, 나머지 여섯 — 코퍼스, 구조, 워크플로, 평가, 피드백, 인터페이스 — 는 도메인을 아는 쪽이 직접 만들어야 한다. 그게 기반 모델이 아무리 좋아져도 수직 팀에게 일이 남는 이유고, 동시에 "지식만 파는" 수직은 위험하다는 경고이기도 하다.
09

사례를 다 보고 남는 것

사례들이 겹치는 지점

  • 검색 전에 압축이 온다. Cerebras는 스레드를 증류하고, DoorDash는 대화를 요약하고, Uber는 질의를 전처리한다. 원문 그대로 임베딩하는 곳이 오히려 드물다.
  • 질의는 두 개의 삶을 산다. 재작성본은 검색으로, 원문은 생성으로(03장). 이 분업을 안 하는 시스템을 찾지 못했다.
  • 권한이 곧 제품이다. Glean·M365·Rovo·Slack AI 전부 "질문자가 볼 수 있는 것만"을 아키텍처 최상단에 놓는다 — 그리고 06장이 보여주듯, 권한의 회수 전파 속도(웹훅 분 단위 vs 전체 크롤 주 단위)가 플랫폼 간 실질 격차다.
  • 단일 스코어러를 믿는 곳이 없다. lexical과 시맨틱은 어디서나 같이 돌고, 융합 방식(RRF, self-tuning 랭커, 관계 신호)만 다르다.
  • 이미 있는 구조를 먼저 쓴다. Rovo는 업무 도구의 관계를, LinkedIn은 티켓의 트리를, Slack AI는 기존 검색 스택을 재활용했다. LLM 추출은 구조가 없을 때의 차선책이다.
  • "아무것도 안 함"이 1급 연산이다. Mem0의 NOOP, Cerebras의 승인 게이트, LlamaIndex의 해시 스킵 — 좋은 지식베이스는 쓰기를 잘하는 만큼 안 쓰기를 잘한다.

직접 짓는다면 — 사례가 주는 순서

  • 질문이 몰리는 채널 하나를 고른다 (Uber도 Cerebras도 여기서 시작했다).
  • 그 도메인 데이터가 가진 구조부터 확인한다. 트리·관계·필드가 있으면 보존하고, 없으면 그때 증류나 추출을 쓴다.
  • v1은 순정 RAG로 빨리 내되, 골든셋 평가를 같이 만든다. Uber의 v1 퇴짜는 평가셋이 있었기에 개선으로 이어졌다.
  • 정확도가 문제로 판명되면 검색 앞뒤(전처리·후처리·가드레일)에 투자한다. 임베딩 모델 교체는 대개 그다음 문제다.
  • 쓰기 경로의 세 갈래(겹침/신규/유사)를 명시적으로 설계한다 — 특히 "비슷한 경우"에 덮어쓰기·LLM 판정·시간적 무효화 중 무엇을 쓸지. 미루면 인덱스가 조용히 썩는다.
  • 회사 전체로 넓힐 때가 오면 그때 권한 동기화를 설계한다 — 나중에 붙이기 가장 아픈 조각이라, 넓힐 계획이 확실하면 처음부터.
