How Companies Actually Run This
Anatomy of Retrieval & Update Journeys

Part 1 dissected the design philosophies of open-source projects; this time we tear down the things that actually run in production. Commercial platforms — Glean, Microsoft 365 Copilot, Atlassian Rovo, Slack AI — plus the in-house builds at Uber, LinkedIn, and DoorDash. This revised edition digs three layers deeper: the full path a user message travels before it becomes a prompt, the three-way update decision when new information arrives, and the vertical-domain playbook for turning AI into a domain expert. 01

Two Camps — Hire a Librarian, or Build the Library

Line up the company cases and they split into two camps. Either way, the underlying structure is the same five-stage pipeline from Part 1. What differs is who owns that pipeline.

The platform camp — connect and go

  • Glean — a standalone search platform with 100+ connectors. The prototype of the commercialized "company brain."
  • Microsoft 365 Copilot — auto-provisions a semantic index per tenant; the largest deployment of the bunch.
  • Atlassian Rovo — builds a Teamwork Graph of 150 billion objects and relationships centered on Jira and Confluence.
  • Slack AI — builds no new index; reuses Slack's existing search stack as the RAG retriever.

The shared battleground: connector breadth and permission sync. They sell "who may see what" as much as they sell search quality.

The in-house camp — dig narrow, go deep

  • Cerebras Knowledge — the system dissected in Part 1 §06. An internal ecosystem on a single Postgres table.
  • Uber Genie — an on-call copilot guarding Slack support channels that field 45,000 questions a month.
  • LinkedIn Support KG-RAG — turned historical tickets into a knowledge graph, cutting resolution time 28.6%.
  • DoorDash Dasher bot — a support chatbot that cut hallucinations 90% with guardrails and an LLM Judge.

The shared battleground: pick one domain and push accuracy to human level. Generality is abandoned from day one.

Postscript: Notion is on this board too. It runs enterprise-search inference for 100 million users on Cerebras hardware, streaming results within 300ms — a signal that the knowledge-base race has moved past retrieval algorithms into an inference-latency fight.

Anatomy of a Question — From User Message to Prompt

The first edition of Part 2 mapped which system runs which journey; this chapter opens every cell of that journey. How a single typed sentence gets processed, and where — in what form — it lands, reconstructed from published pipeline docs and actual prompt source text.

One conclusion up front. In every system surveyed, the user's raw text never goes straight into the search engine. Azure's intent prompt, LangChain's condense prompt, Uber's Query Optimizer, DoorDash's conversation summarization, Cerebras's planner, LinkedIn's entity-and-intent extraction — between the chat box and the index there is always at least one LLM call. At the far end, the opposite holds: the generation model almost always gets the original. The rewrite is a tool for aiming retrieval; the original goes to answer generation with its nuance intact. This division of labor — rewrites for retrieval, originals for generation — is the single most consistent pattern across these systems.

Query transformation tracer — pick an example and follow each stage

* Examples reconstructed from mechanisms described in each system's public docs (Azure intent generation, M365 adjacent-term expansion, the E5 prefix contract, Azure AI Search ACL filter syntax, and so on). The actual internal strings differ per company.

Stage 0 — The Gate: Classifiers Come Before Rewriting

Before the raw text ever reaches an LLM rewriter, it passes a battery of cheap classifiers running in parallel. None of them rewrite — they all judge: is this an attack, is PII mixed in, is this even a question worth answering?

Prompt-attack classification
M365 Copilot runs a jailbreak (UPIA) classifier and an XPIA (cross-prompt injection attack) classifier before model execution to block high-risk prompts. The same capability is sold as parts: Azure Prompt Shields (dual checks on user prompts and documents), AWS Bedrock's PROMPT_ATTACK filter, Google Model Armor, Meta Prompt Guard 2 (a tiny 22M-parameter classifier). Glean publishes numbers for its own gate (Glean Protect): 97.8% detection for direct injection, 90% for indirect.
Trust-boundary tagging — one detail
Bedrock wraps user input in <amazon-bedrock-guardrails-guardContent_xyz> tags and recommends generating the xyz suffix randomly per request. An attack sentence is indistinguishable from system instructions, so you mark "where user input ends" with an unpredictable tag. The docs state explicitly that without the tags, the attack filter does not run at all.
PII masking
Personal data gets substituted before flowing to search or logs. Bedrock's sensitive-information filter masks 31 entity types as {NAME}, {EMAIL}, and so on (mask on input, block on output — policies can differ per direction). The open-source default is Microsoft Presidio. But the warning in Bedrock's own docs stings: CloudWatch invocation logs keep the pre-masking original, verbatim.
Query-type classification → polite refusal
Vertex AI Search's answer API classifies queries as ADVERSARIAL_QUERY, NON_ANSWER_SEEKING_QUERY (chitchat), JAIL_BREAKING_QUERY, and more; on a hit it declines generation and returns a fixed phrase ("A summary could not be generated…") plus the plain search results. The point: the default behavior is feature degradation, not blocking.
The gate is probabilistic. EchoLeak (CVE-2025-32711), disclosed in 2025, bypassed M365 Copilot's XPIA classifier with a single crafted email and reached zero-click data exfiltration. Microsoft itself writes "always add extra validation layers" in its docs. And one more for Korean-language users: Korean is not on Prompt Shields' list of training languages (Model Armor has it). Stage 0 of a Korean-language company brain starts with validating the off-the-shelf parts yourself.

Stage 1 — Rewriting: Conversation to Standalone Query

The last message of a multi-turn conversation is unsearchable on its own ("so how did that go?"). So the first LLM call in every system is the conversion chat history + last message → standalone query. This is the most-copied prompt in RAG history: 

# LangChain CONDENSE_QUESTION_PROMPT — verbatim
Given the following conversation and a follow up question, rephrase the
follow up question to be a standalone question, in its original language.

Chat History:
{chat_history}
Follow Up Input: {question}
Standalone question:
Azure — rewriting is a separately billed item
Azure OpenAI "On Your Data" runs this stage as a separate model call named the intent prompt. The output is not one query but a list of search intents (rewrite and decomposition in one shot). Even the measured cost is public: intent prompt 811–1,385 tokens, output 18–27 tokens. Each intent gets its own search.
DoorDash — summarization, not rewriting
A Dasher's problem arrives scattered across several messages. So instead of "rewrite the last message," it compresses the whole conversation → one sentence of core issue, and that summary becomes the search key. KB articles are likewise split into [issue|resolution] structure with only the issue part embedded, so the match ends up issue-to-issue.
Uber — the corpus catalog goes into the rewrite prompt
EAg-RAG's Query Optimizer polishes vague queries and splits complex ones — and the document-list artifact (titles, summaries, FAQs) is fed in as prompt context. It rewrites knowing "what we actually have." The Source Identifier that follows narrows down which document sets should hold the answer.
The limits
Vertex AI Search's automatic rewriting (maxRephraseSteps) caps at 5 steps. Azure truncates question+history at 2,000 tokens combined. Cerebras has its planner pick tools from "a compressed description of what is in the index" — rewriting and routing fused into a single call.

Stage 2 — Expansion and Decomposition: One Query Becomes Many

One standalone query still is not enough. Embedding similarity misses when vocabulary diverges, and multi-hop questions do not resolve in a single search.

Expansion — same meaning, different words

  • M365 semantic index — search for emails that "praised" the design and adjacent terms elated, excited, amazed join the search scope. Inflections (tech/technology/technologies) and synonyms get absorbed too.
  • Glean — treats expansion as an enterprise-vocabulary problem: acronyms, project names, internal jargon. Its docs name the four rewrite artifacts it logs: spelling correction · synonym expansion · query expansion · intent routing.
  • LangChain MultiQueryRetriever — paraphrase expansion via a "generate 3 different versions of this question" prompt. Routes around the limits of distance-based similarity search through query diversification.
  • HyDE — generate a fake answer document from the question and embed that. Turns hard question→document matching into easy document→document similarity.

Decomposition — one question, several searches

  • Bedrock Knowledge Bases — a single QUERY_DECOMPOSITION flag turns it on. The docs' example: "Who scored more in the 2022 World Cup, Argentina or France?" → two sub-queries, "How much did Argentina score?" + "How much did France score?"
  • Azure — the intent list itself is the decomposition. Per-intent search → chunk collection → merge.
  • Vertex — its docs carry an example splitting "What professions and hobbies do Andie Ram and Arnaud Clément share?" into four sub-queries.
  • LinkedIn KG-RAG — not decomposition but extraction: an LLM pulls named entities (as a Map) and an intent set, used to match graph entry points. The query ultimately gets translated all the way into a Cypher query.

Stage 3 — The Three Bodies of a Query

The moment a rewritten, expanded query heads for the retrievers, it splits into three physical forms. The same sentence becomes an embedding string, a lexical search expression, and a filter expression, fired in parallel — results fused later (RRF and friends).

① The embedding string — prefixes are a contract with the encoder
The E5 family requires the query: prefix on questions and passage: on documents. The maker's FAQ, in essence: that is how the model was trained; skip them and you will see degraded performance. BGE attaches an instruction only on the query side ("Represent this sentence for searching relevant passages:"), Cohere uses an API parameter instead of a prefix (input_type=search_query/search_document), and OpenAI text-embedding-3 attaches nothing. Whether you need a prefix is a property of the encoder, not of your RAG framework.
② The lexical expression — field boosts
The Elasticsearch classic: "fields": ["subject^3", "message"] — a title hit scores 3× a body hit. Glean layers freshness weighting, authority scores, and popularity signals on top, and pushes different fields per intent cohort (ticket IDs hit exact-match fields, policy questions favor canonical sources, how-tos favor the latest procedure docs). Cerebras's rationale is crisp: "when an engineer pastes an error message verbatim, an exact lexical match is almost always the best evidence. No semantic similarity should beat it."
③ The filter expression — permissions, metadata, time
ACLs are not post-retrieval filtering — they are filters attached to the query. Azure AI Search's canonical syntax: group_ids/any(g:search.in(g, 'g1,g2')) — the asker's group list rides along on every query. In M365 the grounding stage itself touches only "content the asker has permission to see," and Slack AI operates "only on data the user can already see." Scoping — Cerebras's project scope, M365's SharePoint library scope — is a species of filter too.
Parallel fan-out
Cerebras: the executor runs the planner's chosen tools in parallel → normalizes into a shared evidence schema. Anthropic's contextual-retrieval reference architecture likewise runs two retrievers per query — BM25 + embeddings — and rank-fuses. The M365 orchestrator fills five candidate-skill slots, lexical match first, then semantic. Azure fires a search per intent. No system on this board finishes with a single search call.

Why the question goes last — the 2023 evidence
Liu et al., Lost in the Middle (2023): performance dips in a U-shape when relevant information sits mid-context. GPT-3.5-Turbo, 20 documents — 75.8% with the answer document at position 1, 53.8% at the middle (position 10), 63.2% at the end (position 20). Middle placement was worse than giving no documents at all (56.1%). Anthropic's prescription ("long material up top, query at the end — up to 30% improvement") survives verbatim in the Claude 5-generation docs as of 2026; Google gives the same advice, and OpenAI recommends the "instructions on both ends" variant.
The 2026 update — the problem is not 'middle' but 'length'
Current models have mostly beaten simple middle-position bias (LongPiBench 2025: "most current models are robust against the lost-in-the-middle issue"). What stays alive is length and interference: NoLiMa (ICML 2025) measured that once lexical overlap is removed, models advertising 128K+ shrink to an effective context of 1–8K; even the Gemini 3 Pro model card admits 26.3% on 8-needle at 1M tokens. Chroma's Context Rot study (18 models): even trivial tasks degrade as input grows. Conclusion — the pipeline that reranks down to top-10 and compresses is now justified not as "middle avoidance" but as context hygiene.
Token budget — the only published formula (Azure)
Question+history ≤ 2,000 tokens · 20% reserved for the response · system message capped per model (400–4,000 tokens; it is appended to an internal base prompt, so overflow gets truncated) · everything left goes to chunks (default 5 × 1,024 tokens). Measured prompt sizes: 3,997–4,854 tokens. Meanwhile Anthropic's contextual-retrieval experiments report top-20 chunks beat 5 or 10 — the budget-versus-quality trade-off has no consensus yet.
Citations — from etiquette to structure
Three generations coexist. ① Prompt directive: "cite every claim" (Azure; uncited documents come back flagged UNCITED_REFERENCE). ② API structure: Anthropic Citations API — pass documents in chunks and validity-guaranteed citation pointers attach to the response, with the quoted text (cited_text) excluded from output-token billing. ③ Post-hoc verification: DoorDash's two-tier guardrail — tier 1 is response-to-evidence semantic similarity (a cheap check); on failure, tier 2 is an LLM evaluation ruling on groundedness and compliance.
Deeper: the practice of chunk formatting and context ordering — XML tags · contextual headers · LongContextReorder

Chunk format. Anthropic's official recommendation is XML structure — per document, <source> and <document_content> inside <document index="n">. The index attribute becomes the citation ID directly. The rationale: when instructions, context, and input mix, XML tags make the boundaries unambiguous. Cerebras cites this doc as a reference for its thread-normalization format.

Contextual headers (contextual retrieval). Before embedding, an LLM generates one or two sentences per chunk answering "where does this chunk sit in the whole document." By Anthropic's measurements, retrieval failure rate drops 5.7%→3.7% (−35%), 2.9% (−49%) with BM25 combined, 1.9% (−67%) with reranking on top.

Ordering. Descending relevance is the default, but LangChain actually ships a transformer called LongContextReorder — from the source docstring: "the less relevant documents will be at the middle of the list and more relevant elements at beginning / end." The algorithm is four lines: reverse the relevance-sorted list, then alternate odd-indexed items to the tail and even-indexed to the head (1st at one end, 2nd at the opposite end, the stragglers in the middle). On 2025 benchmarks, with absolute positional bias reduced in current models, the expected gain has shrunk — a "2023-era mitigation, still shipping" part.

This chapter in one line
A query splits into a retrieval copy (rewrites, expansions, prefixes, filters) and a generation original. Cerebras states it hands the reranker and synthesis LLM the "original question," and Azure ships the "user question and conversation history" verbatim next to the chunks. Retrieval wants a machine-readable query; generation wants the query exactly as the human wrote it — one sentence living two lives is the default design of modern RAG.

Glean

Prototype of the commercial company brain100+ connectors
Founded 2019 · core team of early Google Search engineers · valued around $7B

The reference point for this board. The founders came from Google Search and the approach shows it: they reinvented PageRank for the enterprise to score document "authority," refused to bet on a single embedding — running lexical, vector, and graph signals all together — and then spent a long time grinding on ranking. By their own account, the secret is not algorithmic complexity but the tedious iteration of polishing the ranker.

How it gets built
Connectors scrape three things from every app: content, metadata, and the permission map. Identities from Okta/Azure AD get unified with per-app accounts ("Anthony on Slack" and "Anthony on Salesforce" are the same person), and on top sits an Enterprise Knowledge Graph linking content, people, and activity. Even the embedding model is not used off the shelf — it gets re-refined on that company's language (acronyms, product names, team names).
What it connects
Slack, Google Drive, Confluence, Jira, GitHub, SharePoint, Salesforce, Zendesk and 100+ more · custom sources via a developer API. File parsing covers PDF and Office documents.
Knowledge updates
Tracked on a three-rung ladder: webhooks processed within 5 minutes, incremental crawls per connector from 10 minutes to 6 hours (Salesforce 10 min · Slack/Drive 3 h · Jira 6 h), full crawls typically on a 28-day cycle. Permission changes sync along the same paths. The ranking model is self-tuning, so it gets better for that company as usage accumulates — self-reported: roughly 20% search-quality improvement over 6 months of continuous learning.
Where its permission handling is a class apart
Not an index-time snapshot, not a post-generation filter — ACLs are enforced at query time, every time. That one line is why it can be deployed in regulated industries.

The takeaway
Personalization of the "an engineer is probably not looking for a Salesforce doc" kind is half of search quality. The axis open-source RAG stacks miss — who is asking — is what Glean turned into a first-class ranking signal. That is the moat.

"Zero build cost" is this case's identity — and, conversely, why organizations with messy data get their garbage served right back in search.

Atlassian Rovo · Teamwork Graph

Graph-first150B objects & relations

How it gets built
The core is a unified object model. A Jira issue, an Asana task, a GitHub PR — all normalized into the same "work item" type in one graph, with the relationships between objects wired in (this PR closes this issue; this issue came from the plan in this doc). Not a pile of documents but a map of how work flows.
Retrieval strategy
Relationship traversal comes before vector similarity. "What shipped to Titan this week?" gets answered by stitching merged PRs + in-flight Jira issues + launch-prep docs through their relations. One-line verdict
Part 1's GraphRAG question ("how do you answer queries where the relationship is the answer") solved not with LLM extraction but with the structured relations work tools already have. Extraction cost is zero; the trade is that relationships outside the Atlassian ecosystem stay shallow.

Slack AI

How it gets built — it doesn't
No new knowledge base was created. Slack reuses its existing search ranking stack (lexical + embedding-based ranking ML), tuned over years, as the RAG retriever, unchanged. Message data stays where it always lived; the model just takes those search results and writes the answer.

The In-House Camp in Detail

All three start the same way: the same questions keep landing in one channel. And none of them built general-purpose search. They picked one domain and built retrieval shaped to that domain's data structure.

Uber Genie

On-call copilotAgentic-RAG migration
45,000 questions/month across 154 support channels · 70,000+ answers since 2023-09 · v1: vanilla RAG → v2: EAg-RAG (acceptance relative +27%, wrong answers relative −60%)

It started in the Slack support channels of platform teams like Michelangelo. With questions flooding in and on-call engineers wearing down, they picked RAG over fine-tuning — no reason to shoulder the data-curation and retraining costs.

How it gets built
A Spark batch pipeline scrapes the internal wiki (Engwiki), internal Stack Overflow, and engineering requirement docs, converts them with OpenAI embeddings, and loads a vector DB. Textbook RAG construction — and v1 shipped with exactly that and nothing more.
Where v1 broke
When they attached ~40 security and privacy policy documents and tested against a golden set, the SMEs rejected it. Answers were incomplete, wrong, or missing the needed details. Vanilla chunk-and-embed collapsed on table-heavy policy documents.
The v2 (EAg-RAG) prescription
Agents on both sides of retrieval. A pre-processing agent polishes the query and identifies which document sets should hold the answer; a post-processing agent refines the collected context. Documents got fixed too — a custom Google Docs loader plus LLM-based table extraction and metadata enrichment raised ingest quality itself. In Part 1 terms, they rediscovered Cerebras's distillation, late.
Knowledge updates / platforms
Re-collection and re-embedding by rerunning the batch pipeline. Interface: a Slack bot. Sources: Engwiki · internal SO · ERDs · policy docs.

LinkedIn Support KG-RAG

Tickets → knowledge graphSIGIR 2024
6 months in production · retrieval MRR +77.6% · median per-issue resolution time -28.6%
The problem statement came first
Chunk historical support tickets as plain text and you shatter the ticket's structure. A ticket is natively a tree of title, symptoms, cause, and resolution, and tickets link to each other through relations like "same root cause" or "follow-up issue." That structure is the retrieval signal — and chunking grinds it away.
How it gets built
Convert the entire ticket history into a knowledge graph. Inside each ticket, keep the tree as nodes (intra-issue); wire tickets to each other as links (inter-issue); embed every node as a semantic-search entry point.
Knowledge updates
New tickets attach onto the graph. The paper itself leaves automatic graph-template extraction and query-driven dynamic updates as future work — refreshingly honest, as in-house system papers go.
Where it meets Part 1
Unlike GraphRAG, no LLM extracted these relations — the graph came from preserving structure the data already had. Same vein as Rovo, at a fraction of the cost. The lesson: if your domain data has structure, preservation beats extraction.

DoorDash Dasher Support Bot

Guardrails-firstThousands of Dashers served daily
Replaced a flow-based bot · hallucinations -90% · severe compliance issues -99%
How it gets built
Two source types: existing support knowledge-base (KB) articles, and transcripts of previously resolved cases. The focus was making already-validated assets searchable rather than writing new documents. Since Dashers use the bot mid-delivery, multilingual support and response latency were design constraints too.
The heart of retrieval — summarize first
A Dasher's problem arrives scattered across messages. So before any search, the whole conversation is compressed into a one-sentence core issue; that summary finds similar past cases, and each case drags along its mapped KB articles. The search key is not the question — it is the "summarized problem."
The quality triad
After generation, a Guardrail checks every response for groundedness and compliance in real time, handing failures to a human agent. A separate LLM Judge scores continuously on five metrics — retrieval accuracy, response accuracy, grammar, coherence, relevance — and those results feed back into the improvement pipeline.
Knowledge updates
KB articles are the single source of truth, so fix the article and the bot follows. Newly resolved cases accumulating in the similar-case pool is the second update axis.

The Three Forks of Updating — When New Information Arrives

If the retrieval journey is the read path, this chapter is the write path. When new information arrives, the system compares it against existing knowledge and routes it down one of three forks: it overlaps, it does not, or it is similar-but-different. Each fork has real mechanisms behind it, with names and thresholds on the public record.

Before the fork there is a gate zero — identity checks that finish cheaply, before any LLM: Cerebras drops duplicate deliveries via Slack's unique event ID; the LlamaIndex ingestion pipeline skips "same document, unchanged" via a doc_id + content-hash map; Google's crawler treats a 64-bit SimHash within Hamming distance ≤ 3 as a near-duplicate (a parameter validated at multi-billion-page scale); and SemDeDup, the standard for training-data dedup, calls cosine similarity ≥ 0.99 (ε=0.01, the NVIDIA NeMo default) a semantic duplicate. Only candidates that pass this gate — i.e., are not obvious duplicates — enter the decision tree below.

New information arrivesSlack thread · document edit · resolved case · extracted fact
Compare against existing knowledge — similar-item search (Mem0: top-10 similar memories per candidate fact)

The Adjudicator in the Flesh — Mem0's Two Phases

The most concrete public version of "compare, then route three ways" comes from the memory system Mem0 (2025 paper + open source). In the extraction phase an LLM pulls candidate facts from the conversation; in the update phase, each candidate gets its top-10 similar memories via vector search and is put in front of an LLM through a function-calling interface — four options: ADD / UPDATE / DELETE / NOOP. This chapter's three forks, reified as four tools. The headline numbers: p95 latency down 91% versus full-context (17.1s → 1.44s), token costs cut 90%+ — but the competitive comparisons come from Mem0's own paper, so trust them exactly that far.

The Problem at Scale — The Connector Sync Ladder

In document-level systems, the three forks are implemented as connector sync. The pattern is the same three-rung ladder everywhere: webhooks (minutes) → incremental crawls (minutes to hours) → full crawls (weekly-ish, the final consistency guarantee). When a Conflict Survives to Query Time — Ranking Adjudicates

Contradictions the write path failed to catch (old policy doc vs new policy doc) end up refereed by ranking. Glean's signal stack: freshness weighting + authority score + usage signals. The balance they articulate in the docs is good — "an over-tuned recency boost lifts new-but-wrong documents above the canonical policy page," and "a WiFi-password doc untouched for years is still authoritative if people keep finding it." Cerebras solves the same problem by mixing age decay into RRF: on a tie, the newer thread wins. Staleness that resists automatic adjudication goes back to humans: Glean Verification puts re-verification reminders on document owners, and stale docs get the Deprecated badge — kept in search, flagged, not deleted.

Production in the Flesh — How DoorDash Turns Resolved Cases into Knowledge

DoorDash is the only company to publish, at company scale, the full journey of new information entering a knowledge base (2025). Escalation to a human agent is itself treated as the bot's failure signal, and those conversations become knowledge candidates.

The eight-stage pipeline
Select only escalated conversations from the anonymized pool → embed, then cluster online (each new conversation compared by cosine to existing cluster centroids; below a 0.70–0.90 threshold, spawn a new cluster) → human review of top clusters (manual merge of duplicate clusters) → classify clusters as "actionable problems vs informational questions" → for informational ones, an LLM drafts an article from the issue summary + exemplary resolutions → content-expert review (policy accuracy, tone, edge cases) → embed and deploy approved articles → serve through the existing RAG.
Results — read with the caveats attached
Escalation rate on high-traffic clusters: control 78% → treatment 43% (A/B experiment). About 75% of treatment-group KB retrieval events returned only content this pipeline created (UGC-derived). Not a global number — high-traffic clusters only, period and volume undisclosed: the classic limits of company-blog figures. Also: DoorDash has a separate −90% — the 2026 simulation flywheel cut hallucination rates ~90% pre-deployment, a different value from the 2024 guardrail's production −90%. Do not conflate them.
Mapping to the three forks
Fork B (new-knowledge discovery and drafting) is done by machine. Fork A (overlap) is the cluster threshold + manual merges. But Fork C — contradiction with existing articles — has no automation: no conflict-check mechanism between generated and existing KB articles appears anywhere in the public material; human review doubles as that layer.
Contrast with memory systems
Mem0, Graphiti, and Letta automate all three forks — at the scale of one user's memory. DoorDash automates detection and drafting but puts humans in front of the write — because this is knowledge the whole company sees. The variable is not the technology but the blast radius of a badly written fact.
This chapter in one line
The practice of updating is less "fix the knowledge" and more design the forks. Overlap → discard (as an explicit operation, no less); new → append without global recomputation; similar → pick whichever of the three schools (overwrite / LLM adjudication / temporal invalidation) fits the data. And what to automate versus what to put behind a human is decided by blast radius, not accuracy.

Vertical Domains — How to Make an AI a Domain Expert

Push the company-brain playbook (ingest → structure → retrieve → synthesize → verify) wholesale into one professional field and you get vertical-domain AI. Harvey in law, Hebbia in finance, OpenEvidence and Abridge in medicine — and in fact this report's in-house cases are all verticals too. Cerebras KB's vertical is semiconductor engineering; DoorDash's is delivery-logistics support.

To make the cases comparable, we decompose them into seven levers: ① corpus (did you secure domain data experts already trust — licensed or self-accumulated), ② structure (did you impose the domain's ontology and schema on the data), ③ workflow (did you engrave the expert's task decomposition into the product), ④ evaluation (do you measure against an expert-built golden set or benchmark), ⑤ feedback loop (do expert corrections flow back into the system), ⑥ model strategy (fine-tune, retrieve, or both), ⑦ interface (did you move into the tools experts already live in). Click through the tabs for per-case profiles — the lever bars are qualitative reads of the public record.

Fine-Tune or Retrieve — Tuition the Industry Already Paid

Cautionary tale — BloombergGPT (2023)
Trained a 50B model from scratch on a proprietary 363B-token financial corpus (est. $10M+). The outcome is famous — general-purpose GPT-4 won on most financial tasks. What Bloomberg actually shipped was not that model but retrieval-grounded products inside the Terminal: Document Insights over 200M+ documents, transparency links jumping to source excerpts. The strategy "train your own domain GPT" lost to "ground a frontier model in your own corpus."
The exception clause — when a gradable rulebook exists
Ambience Healthcare built an ICD-10 medical-coding model with OpenAI RFT (reinforcement fine-tuning) and beat a panel of 18 board-certified physicians by a relative 27% (evaluated against a golden panel labeled by consensus of 4+ specialists). Why did it work here — ICD-10 is a narrow task with a verifiable rulebook and gradable answers. Harvey's case-law model (10B tokens of mid/post-training, 97% of lawyers preferring it over GPT-4 — announced by OpenAI itself) fits the same pattern. Open-ended understanding goes to retrieval; closed-book grading goes to fine-tuning.

Vertical vs Horizontal — Both Sides' Logic

Why vertical wins (the bull case)

  • The market is a different size — vertical AI attacks labor and services budgets, not IT budgets (Bessemer: "10× the TAM of traditional software markets").
  • They own the yardstick — every benchmark that matters came from a vertical company. Harvey's BigLaw Bench and LAB (1,200+ tasks, 75,000+ expert-written rubric criteria, all-pass grading where one miss fails the task); Sierra's τ-bench (GPT-4o-class agents succeed under 50% at policy-compliant customer service; pass^8 reliability over 8 retries collapses below 25% in the retail domain).
  • Accountability machinery — OpenEvidence refuses to answer when evidence is insufficient, Abridge traces every note sentence back to source audio (Linked Evidence), Sierra bills only for resolved cases. A layer frontier models do not ship by default.
  • The data loop — EvenUp's accumulated settlement-and-verdict DB, Abridge's weekly model updates (folding in physician corrections), Sierra's failure mining. Outcome data no license can buy.

Why horizontal absorbs it (the bear case)

  • Foundation models are absorbing the knowledge layer — Harvey's own data is the evidence: on BigLaw Bench, base-model scores jumped from ~60% (GPT-4o/Claude 3.5) to ~90% (GPT-5/Claude 4.5/Gemini 3) in two generations.
  • The labs move in directly — OpenAI's healthcare triple: ChatGPT Health (2026-01-07, consumer) → ChatGPT for Healthcare (01-08, hospital enterprise) → ChatGPT for Clinicians (04-23, free for NPI-verified physicians) — aimed squarely at OpenEvidence's position three months after its $12B round. On HealthBench Professional, ChatGPT for Clinicians (GPT-5.4-based) scores 59.0 and GPT-5.6 Sol 60.5, ahead of physician responses (43.7, under unlimited-time + web-search-allowed conditions) — with the permanent footnote that this is OpenAI grading OpenAI.
  • Horizontal platforms ship vertical packs — Glean layered on an agent library with 30+ prebuilt agents for sales, engineering, IT, and HR. Incumbents holding the interface loom too — Epic walked into Abridge's front yard with its own "Art for Clinicians."
Net: knowledge (memorized domain common sense) is being absorbed by foundation models, at a pace the vertical companies' own benchmarks prove. Where verticals keep winning is five things: closed-universe work product (real deliverables graded all-pass), outcome-data loops, accountability and audit machinery, living inside the expert's tools (Word, Epic, the Terminal, Slack), and outcome-based pricing. And the two camps interpenetrate as they fight — Harvey went in as an agent inside M365 Copilot, and Sierra agents surface in ChatGPT.

Applying It to Your Domain — The Order the Cases Agree On

1–5: Laying the foundation

  • 1. Inventory billable units of work, not topics. Harvey built BigLaw Bench from lawyers' actual time entries. Start by writing down 10–20 tasks experts get paid for (or worn down by) as "instructions + a closed document universe + expected deliverable."
  • 2. Secure a corpus experts already trust. License it (OpenEvidence × NEJM/JAMA) or accumulate it yourself (Abridge's 1.5M consented clinical conversations). Either way, it is not scraping.
  • 3. Impose domain structure before embedding. SOAP notes, ICD-10, regulatory decision trees (Norm Ai), ticket trees (LinkedIn), Cerebras's {question/summary/resolution/system/code-refs} schema — structure is what turns retrieval into expertise.
  • 4. Engrave the expert's task decomposition as workflow. Not a mega-prompt — Hebbia's ISD steps, Sierra's Journeys, EvenUp's assembly line.
  • 5. SME golden set before scaling, graded all-pass. Uber's v1 survived because there was a golden set to reject it.

6–10: Scaling without losing it

  • 6. Default to retrieval + a frontier model. Fine-tune only where a gradable rulebook exists. BloombergGPT paid the tuition for everyone; Ambience's RFT showed the exception's preconditions. Per-subtask multi-model routing (Harvey's Auto) has become standard too.
  • 7. Move into the expert's existing screen. A Word add-in, Epic's first "Pal," a Terminal panel, a Slack bot, MCP into Claude Code. A new destination app is the last resort.
  • 8. Make every answer auditable. Linked Evidence (note→audio), transparency links (summary→source excerpt), citations (claim→paper), source scores.
  • 9. Feed expert corrections back on a fixed cadence. Abridge turns physician edits into weekly updates; EvenUp turns its 100+ in-house experts' reviews into model improvements.
  • 10. Design the refusal path as carefully as the answer path. OpenEvidence stays silent on insufficient evidence, DoorDash hands guardrail failures to a human, Sierra does not bill failed cases. In expert domains, a safe non-answer is trust itself.
This chapter in one line
"Making an AI a domain expert" is not a model swap — it is working seven levers at once. The foundation model does exactly one of them for you (the sixth, the model); the other six — corpus, structure, workflow, evaluation, feedback, interface — must be built by whoever knows the domain. That is why vertical teams keep having work no matter how good foundation models get — and also the warning that a vertical selling only knowledge is in danger.

What Remains After All the Cases

Where the cases converge

  • Compression comes before retrieval. Cerebras distills threads, DoorDash summarizes conversations, Uber pre-processes queries. Embedding raw text is the exception, not the rule.
  • A query lives two lives. Rewrites go to retrieval, originals to generation (§03). We found no system that skips this division of labor.
  • Permissions are the product. Glean, M365, Rovo, and Slack AI all put "only what the asker can see" at the top of the architecture — and as §06 shows, the real gap between platforms is revocation propagation speed (webhook minutes vs full-crawl weeks).
  • Nobody trusts a single scorer. Lexical and semantic run together everywhere; only the fusion method differs (RRF, self-tuning rankers, relationship signals).
  • Existing structure gets used first. Rovo reused work tools' relations, LinkedIn reused ticket trees, Slack AI reused its own search stack. LLM extraction is the fallback for when there is no structure.
  • "Do nothing" is a first-class operation. Mem0's NOOP, Cerebras's admission gate, LlamaIndex's hash skip — a good knowledge base is as good at not writing as it is at writing.

If you build one — the order the cases suggest

  • Pick one channel where questions pile up (both Uber and Cerebras started there).
  • Check what structure the domain data already has. Trees, relations, fields — preserve them if present; reach for distillation or extraction only if not.
  • Ship v1 as vanilla RAG, fast — but build the golden-set eval alongside it. Uber's v1 rejection turned into improvement precisely because the eval set existed.
  • When accuracy proves to be the problem, invest around retrieval (pre-processing, post-processing, guardrails). Swapping the embedding model is usually the later problem.
  • Design the write path's three forks (overlap/new/similar) explicitly — especially which of overwrite, LLM adjudication, or temporal invalidation handles the "similar" case. Postpone it and the index rots quietly.
  • Design permission sync when it is time to go company-wide — it is the most painful piece to bolt on later, so if company-wide is certain, do it from day one.
